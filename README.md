# Item-wise Gross Profit Analysis — Grocery Retail

Turning one month of raw POS sales data (3,600+ SKUs) into a margin strategy for an independent grocery store: which items to push, which to bundle, and which are quietly losing money.


---

## The problem

The store's billing software exports a flat "item-wise gross profit" report every month — thousands of rows, no structure, no way to tell at a glance which of the 3,600+ SKUs were actually profitable versus which were moving volume at breakeven or a loss. Pricing and stocking decisions were being made without that visibility.

## What this project does

1. **Cleans and processes** the raw POS export in Python/pandas — strips report banners, removes inactive SKUs, recomputes GP% from first principles rather than trusting a pre-formatted column.
2. **Segments every item** into a margin tier (High ≥15% / Mid 8–15% / Low 0–8% / Loss-making <0%), ranked by rupee impact within each tier.
3. **Surfaces the findings** in an interactive dashboard, with a recommended action per item.
4. **Translates the analysis into a merchandising strategy** — 8 priced product bundles that pair low-margin footfall items (rice, oil, sugar) with high-margin ones (spices, dry fruits) to lift blended margin, complete with ready-to-send sales messages.

## Results

| Metric | Value |
|---|---|
| SKUs analyzed | 3,668 active items (of 3,871 total; 203 had zero movement in the period) |
| Total sales | ₹1.57 Cr |
| Total gross profit | ₹14.5 L |
| Blended GP% | 9.21% |
| Loss-making items identified | 105 |
| Bundle packs designed | 8, with blended margins from 14% to 38% |

## Methodology

**Data cleaning.** The raw export has a 5-row report banner, a totals row at the bottom, and item names prefixed with `*` for items that moved during the period. `analysis/gross_profit_analysis.py` strips all of this and standardizes column names (`Amt. Out` → `sales`, `Gross Profit` → `gross_profit`, etc.).

**Handling inactive stock.** 203 of 3,871 SKUs had zero sales in the period (no opening/closing movement) — these are excluded from the margin analysis, since GP% is undefined (not zero) for an item that didn't sell.

**GP% is recomputed, not copied.** The export includes a pre-formatted "GP Ratio" text column; the script ignores it and recalculates `gross_profit / sales × 100` directly from the numeric columns, so every figure in the dashboard is independently reproducible from the raw file.

**Tiering logic.**
- **High margin (≥15%)** — profit engines; feature at the counter, push on WhatsApp
- **Mid margin (8–15%)** — reliable bundle anchors
- **Low margin / high volume (0–8%)** — footfall drivers (sugar, ghee, oil); never sell alone, always attach a high-margin item
- **Loss-making (<0%)** — pricing needs to be fixed or the item dropped

Within each tier, items are ranked by absolute ₹ gross profit (or ₹ loss) rather than GP% alone, so the dashboard surfaces the items that matter most to the bottom line, not just the highest percentages on low-volume items.

**From analysis to action.** The bundle catalogue (`dashboards/bundle_offers.html`) was built directly from the tiered output: each pack anchors on a mid/low-margin staple (rice, dal, atta) and attaches 4–8 high-margin items (spices, dry fruits) to lift the blended margin of the whole pack well above any single low-margin item on its own.

## Running the analysis

```bash
cd analysis
pip install -r requirements.txt
python gross_profit_analysis.py
```

This reads `data/item_wise_gross_profit_apr2026.xlsx` and regenerates every CSV in `/outputs` — the same tiered lists that drive the dashboards.

## Deploying to GitHub Pages

1. Push this repo to GitHub.
2. Repo **Settings → Pages → Deploy from a branch**, select `main` and `/ (root)` (or `/dashboards` if you'd rather serve that folder directly).
3. Your live case study will be at `https://YOUR-USERNAME.github.io/grocery-retail-gp-analysis/dashboards/index.html`.
